# Bay Ave Barnacle — Flood Prediction Model v0.9

**Target:** Predict whether 342 Bay Ave (Highlands NJ) will flood in
the next 24 hours, with per-landmark depth estimates at 18 named
landmarks, PLUS a first-generation pluvial (rain-driven) flood
estimate for the event class the tide-keyed model cannot represent.

**Promotion date:** 2026-07-06 (the day of the pluvial flash flood
that answered HANDOFF item 14).

**Changes from v0.8:**

1. **Pluvial model, v0.9-alpha** (`estimate_pluvial_water`) — first
   model for rain-driven flooding, surfaced as scenario estimates in
   the pluvial advisory banner (categorical + depth scenarios), NOT
   as a change to the tide-keyed prediction path.
2. **QPF input fixed** — rain had been silently 0.0 in every
   production run ever (NWS moved QPF out of forecastHourly; only
   the raw gridpoint endpoint has it). `fetch_nws_qpf()` is the new
   source; the tide+rain term now actually receives rain.
3. **Pluvial flood-risk advisory** — categorical banner
   (possible/elevated) from QPF rate / cumulative / PoP +
   thunderstorm wording, with v0.9-alpha scenario depths.
4. **Porch ladder re-anchored from real measurements** (7/6 event
   cross-fits + taped risers, `assets/porch-measurements.txt`):
   - `lawn_step`: 4.58 → **4.66** (triple-anchored)
   - `porch_step` (5.08) **removed** — it corresponded to no physical

## Core formula (tide-keyed path — unchanged from v0.8)

```
LOCAL_ENHANCEMENT = 0.00 ft
MLLW_TO_NAVD88     = -2.82 ft

water_at_342_NAVD88 = Sandy_Hook_MLLW + LOCAL_ENHANCEMENT + MLLW_TO_NAVD88

if peak_rain_rate_in_hr > 0.1:          # rain in [-90min, +15min] of peak
    water_at_342_NAVD88 += (8.0 * tanh(peak_rain_rate_in_hr)) / 12.0

depth(landmark) = max(0, water_at_342_NAVD88 - landmark.elevation) * 12
```

Wind adjustment (v0.8, unchanged): offshore forecast winds at peak
(S/SSW/SW/WSW/SSE) → report a −0.13 ft "expected actual" line
alongside the main prediction.

## Pluvial model (NEW — v0.9-alpha)

```
base = max(bay_water_navd88, 3.52)         # street bowl bottom (grate_SW)
drains functional (bay < 3.52): lift = 1.40 · tanh(rate / 1.2)
drains blocked   (bay ≥ 3.52): lift = 8 · tanh(rate) / 12   # the v0.8 term

pluvial_water = base + lift     (rate > 0.1 in/hr)
```

Calibration (2 events):

| Event | Bay water | Rain rate | Model | Observed |
|---|---:|---:|---:|---|
| 2026-07-06 pure pluvial | ~2.6 | ~1.7 in/hr burst (FITTED — observed rate unknown) | 4.76 | 4.77–4.84 (peak band) |
| 2025-10-30 compound | 4.81 (SH 7.63) | 1.45 in/hr (recorded) | 5.41 | ≥ 5.25 |

Why free-drain lift exceeds blocked-drain lift: bowl-vs-plain
geometry (see code comment). Known warts: regime discontinuity at
bay = 3.52; the 1.7 in/hr burst is a fitted assumption; no timing
model (the banner says "a burst today could do X," not when).
Stage-storage curve + observed rain input are the v0.9-final work.

Surfaced ONLY in the pluvial advisory banner as ±3″-class scenario
estimates; the tide-keyed table/JSON are untouched by it.

## What might break this model

1. **Pluvial timing is unmodeled** — the banner is categorical.
2. **The 1.7 in/hr burst is fitted, not measured** — get observed
   rain for 7/6 (MRMS/Rutgers) to pin it.
3. **QPF smears convective bursts** (~6-h buckets) — PoP + wording
   triggers carry the banner; radar nowcast is the real fix (9e.2).
4. **Regime discontinuity at bay = 3.52** in the pluvial form —
   stage-storage curve needed.
5. **Seasonality CSV thresholds are v0.6-era** — annual analytics
   refresh should re-run against v0.9 thresholds.
6. Wind rule, enhancement extrapolation above SH 7.3, cold lockout,
   SLR drift: carried from v0.8 unchanged.
